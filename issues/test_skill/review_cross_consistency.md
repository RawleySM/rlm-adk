## 3. Missing Understanding

Things no agent covered that the implementor needs:

### 3.1 How `enabled_skills` in fixture config maps to skill loading

The fixture's `config.enabled_skills` (e.g., `["test_skill"]`) is read by `instrumented_runner.py:720`:
```python
_enabled_skills = router.config.get("enabled_skills") or None
```
This is passed to `create_rlm_app(enabled_skills=_enabled_skills)`, which sets it on the `RLMOrchestratorAgent`. Inside `_run_async_impl()`:
- `collect_skill_repl_globals(enabled_skills=self.enabled_skills or None)` discovers and imports skill modules matching enabled names
- `load_adk_skills(self.enabled_skills)` loads Skill objects for SkillToolset (only when `self.enabled_skills` is truthy)

**For children**: `create_child_orchestrator()` does NOT pass `enabled_skills`, so the child's `enabled_skills` defaults to `()`. But `() or None` evaluates to `None`, meaning `collect_skill_repl_globals(enabled_skills=None)` discovers ALL skills. So children get all skill REPL functions but no SkillToolset.

### 3.2 How `initial_state` gets injected into the session

In `instrumented_runner.py:764-768`:
```python
initial_state = router.config.get("initial_state") or None
session = await session_service.create_session(
    app_name="rlm_adk", user_id="test-user", state=initial_state,
)
```
The `initial_state` dict from the fixture config is passed directly to `session_service.create_session()`. This means keys like `user_provided_ctx`, `repo_url`, `root_prompt`, `skill_instruction` are pre-loaded into `session.state` BEFORE the orchestrator runs. The orchestrator then reads them in `_run_async_impl()` via `ctx.session.state.get(USER_PROVIDED_CTX)` (Path B at line 453).

### 3.3 What happens to `DYN_SKILL_INSTRUCTION` during child dispatch

1. Parent orchestrator sets `DYN_SKILL_INSTRUCTION` in initial state (line 411) via `instruction_router(self.depth, self.fanout_idx)`
2. When child dispatch occurs, the child orchestrator overwrites `DYN_SKILL_INSTRUCTION` with its own value (if instruction_router is configured)
3. After child returns, `post_dispatch_state_patch_fn()` (dispatch.py:537-546) restores the parent's `_parent_skill_instruction` to `DYN_SKILL_INSTRUCTION`
4. REPLTool calls this patch function after each code execution (repl_tool.py:261-263)

No agent fully traced this restore-after-child-dispatch cycle.

### 3.4 How `post_dispatch_state_patch_fn` works exactly

From `dispatch.py:537-546`:
```python
def post_dispatch_state_patch_fn() -> dict[str, Any]:
    delta: dict[str, Any] = {}
    if _parent_skill_instruction is not None:
        delta[DYN_SKILL_INSTRUCTION] = _parent_skill_instruction
    return delta
```
It returns a dict with at most one key (`DYN_SKILL_INSTRUCTION`). REPLTool applies this by writing each key to `tool_context.state` (repl_tool.py:261-263). This is called in three places: normal completion, CancelledError handler, and Exception handler.

### 3.5 How `_collect_completion` works exactly

1. `_rlm_terminal_completion` attr on reasoning_agent (set by worker_retry after_tool_cb)

The output is a `CompletionEnvelope` with `display_text` used as the final answer.

### FIFO queue order verification

The responses are consumed in exact sequential order [0, 1, 2, 3, 4, 5, 6, 7] because:
- Call #0: Root reasoning makes first API call
- Call #1: Child d1 runs synchronously (root REPL thread is blocked), so child d1's reasoning makes the next API call
- Call #2: Grandchild d2 runs synchronously (child d1's REPL thread is blocked), so grandchild d2's reasoning makes the next API call
- Call #3: Grandchild returns, child d1's REPL resumes, child d1's reasoning sees tool result and makes next API call
- Call #4: Child d1 returns, root REPL resumes, root reasoning sees tool result and makes next API call
- Call #5: Root REPL thread is blocked during llm_query_batched. Batch child 0 starts first (task order in `asyncio.gather`), makes API call
- Call #6: Batch child 1 makes API call (may interleave with #5 in theory, but in practice with fake server and single event loop, task 0 completes before task 1 starts its API call)
- Call #7: Both batch children return, root REPL resumes, root reasoning makes final API call

**Critical concurrency note for responses #5 and #6**: Because `asyncio.gather()` creates tasks in order and the fake server has zero network latency, batch child 0 runs to completion before batch child 1 starts. This is because:
1. Both tasks are created but only progress when awaited
2. `_run_child` acquires the semaphore (non-blocking when limit=3)
3. Child 0's `reasoning_agent.run_async()` runs first in the event loop
4. Child 0 makes its API call, gets response #5, processes it, completes
5. Event loop returns to child 1, which then makes its API call, gets response #6

This deterministic ordering is reliable in the single-threaded asyncio event loop with zero-latency fake server. For fixtures with slow operations or real network, ordering could differ.
