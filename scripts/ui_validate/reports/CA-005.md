        ## Validation Report: CA-005

### Summary
**FAIL** — The `return_value` + `down` arrow combination is never produced by any code path in `flow_builder.py`. The renderer generically supports it, but this manifest item describes unreachable behavior.

### Evidence

1. **`flow_builder.py` — all three `FlowArrow` constructions (lines 134, 148–153, 176–182):**
   - Line 134: `FlowArrow(direction="down", arrow_kind="execute_code", label="execute_code")` — down + execute_code
   - Lines 149–153: `FlowArrow(direction="right", arrow_kind="llm_query", ...)` — right + llm_query
   - Lines 177–181: `FlowArrow(direction="left", arrow_kind=result_kind, ...)` where `result_kind` is `"return_value"` (no error) or `"set_model_response"` (error) — left + return_value/set_model_response

   **No `FlowArrow` construction in the codebase combines `direction="down"` with `arrow_kind="return_value"`.**

2. **`flow_models.py` — type definitions (lines 16–18):**
   - `ArrowDirection = Literal["down", "right", "left"]` — `"down"` is a valid direction
   - `ArrowKind = Literal["execute_code", "llm_query", "set_model_response", "return_value"]` — `"return_value"` is a valid kind
   - The `FlowArrow` dataclass (line 73) accepts any combination of these types, so a `down` + `return_value` arrow is type-valid but never constructed.

3. **`flow_connectors.py` — renderer behavior (lines 37–38):**
   - Line 37: `color = _ARROW_COLORS.get(arrow.arrow_kind, "var(--text-1)")` → `"return_value"` maps to `"var(--accent-active)"` (line 22)
   - Line 38: `unicode_arrow = _ARROW_UNICODE.get(arrow.direction, "↓")` → `"down"` maps to `"↓"` (line 14)
   - The renderer is a straight-through function with no direction/kind validation or early returns — it **would** render `down` + `return_value` correctly as a `↓` arrow in `var(--accent-active)` color if ever given that input.

4. **No other producer of `FlowArrow` exists in the codebase.** `flow_builder.py:_process_node` is the sole call site (confirmed via the imports and `build_flow_transcript` entry point).

### Bugs Found

- **Bug:** Manifest item CA-005 ("return_value arrow (down)") describes a `FlowArrow` combination that no code path ever produces. The `return_value` arrow kind is only emitted with `direction="left"` (line 175–181), and `direction="down"` is only emitted with `arrow_kind="execute_code"` (line 134). This makes the manifest item a phantom — it describes a renderer capability, not an observable UI element.
- **Location:** Manifest definition (external to codebase) + `flow_builder.py` lines 134, 177
- **Severity:** warning
- **Fix:** Either (a) remove CA-005 from the manifest since no production data path produces this combination, or (b) reclassify it as a "renderer capability test" rather than a "UI element validation," or (c) if a downward return-value arrow is intended (e.g., for a future non-child return flow), add the corresponding `FlowArrow(direction="down", arrow_kind="return_value", ...)` construction in `flow_builder.py`.

### Validation Notes

Static analysis of all three source files confirms the renderer (`render_flow_arrow`) handles any direction/kind combination generically via dict lookup, so a synthetic `FlowArrow(direction="down", arrow_kind="return_value")` would render as `↓` in `var(--accent-active)` — the dicts and render logic are unambiguous. However, `flow_builder.py` is the sole producer of `FlowArrow` blocks and it never emits this specific combination: `return_value` arrows always go `left` (child result returning to parent), and `down` arrows always carry `execute_code` (REPL submission). The manifest item therefore describes dead/unreachable UI behavior. This validation is limited to static analysis — no runtime rendering was verified.
