**PASS** — The active depth metric chip is fully implemented across all three pipeline stages (writer → controller → renderer), with correct data flow from pane depth to UI label.

   - `live_models.py:268` — `active_depth: int = 0` field on `LiveRunStats` dataclass (frozen)
   - `live_models.py:294` — `active_depth: int` field on `LiveRunState` dataclass (frozen)

   - `live_app.py:548-555` — Wrapping `div` with pill styling (`border-radius: 999px`), contains `ui.label(label)` (muted color, 0.72rem) and `ui.label(value)` (primary color, 0.74rem)

### Bugs Found
None.

### Validation Notes
All three pipeline stages (writer, controller, renderer) were verified by reading the actual source code. The `LiveRunStats.active_depth` field is computed in `_build_snapshot()` from the deepest/most-recently-active pane's `.depth` attribute, propagated unchanged through `_refresh_run_state()`, and rendered via `_metric_chip("active depth", ...)` using two `ui.label` elements styled as a pill-shaped chip. The fallback to `0` when no active candidate exists is confirmed at `live_loader.py:819`. The only caveat is that active candidate selection prioritizes recency over depth (depth is the tiebreaker), so `active_depth` reflects the most recently active pane's depth rather than strictly the maximum depth — this is correct behavior for showing *current* active recursion depth.
