        ## Validation Report: SK-027

### Summary
**PASS with WARNING** — `APP_MAX_DEPTH` is correctly defined, in `EXPOSED_STATE_KEYS`, enforced as a depth guard in `dispatch.py`, persisted by `sqlite_tracing.py`, and present in replay fixtures. However, the `app:max_depth` session state value is **never read by the dispatch depth guard** — the guard uses the `max_depth` function parameter default (5) + `RLM_MAX_DEPTH` env var override, creating a latent disconnect between the state key and runtime behavior. The test plan also contains one constructor bug.

### Evidence

1. **Constant definition** — `state.py:12`: `APP_MAX_DEPTH = "app:max_depth"` — exact string value confirmed.

2. **EXPOSED_STATE_KEYS membership** — `state.py:136`: `APP_MAX_DEPTH` is in the `EXPOSED_STATE_KEYS` frozenset (lines 131–142). It is **not** in `DEPTH_SCOPED_KEYS` (lines 145–159), correcting the selector report's inaccuracy.

3. **dispatch.py depth guard** — Three `max_depth` references confirmed:
   - Line 115: `max_depth: int = 5` (default parameter)
   - Line 140: `max_depth = int(os.getenv("RLM_MAX_DEPTH", str(max_depth)))` (env var override)
   - Lines 281–289: `if depth + 1 >= max_depth:` returns `LLMResult(..., error=True, error_category="DEPTH_LIMIT")`

4. **sqlite_tracing.py persistence** — Lines 662–663: `if state.get("app:max_depth") is not None: config["max_depth"] = state["app:max_depth"]` — reads the state key for config persistence.

5. **repl_capture_plugin.py snapshot filter** — Line 60: includes `"app:max_depth"` in its filter list.

6. **Replay fixture** — `recursive_ping.json:4`: `"app:max_depth": 5` present in initial state.

7. **State key NOT wired to dispatch** — `orchestrator.py:279–288` calls `create_dispatch_closures()` **without** a `max_depth=` argument. Neither `orchestrator.py` nor `agent.py` contain any reference to `app:max_depth` or `APP_MAX_DEPTH`. The dispatch guard always uses the hardcoded default `5` (or `RLM_MAX_DEPTH` env var), ignoring the session state value entirely.

8. **Test plan constructor bug** — The test plan calls `DispatchConfig(main_model='fake-model', ...)` but `DispatchConfig.__init__` takes `default_model` (dispatch.py:92), not `main_model`. This would raise `TypeError` at runtime.

### Bugs Found

- **Bug: `app:max_depth` state key is not read by the depth guard**
  - Location: `rlm_adk/orchestrator.py:279–288` (missing `max_depth=` kwarg); `rlm_adk/dispatch.py:115` (unused state key)
  - Severity: **warning**
  - Description: The `app:max_depth` session state value (set via replay fixture `initial_state` or eval runner) has **no effect** on the dispatch depth guard. The guard uses the `max_depth` parameter's hardcoded default of `5`, optionally overridden by the `RLM_MAX_DEPTH` env var. If a user sets `app:max_depth` to `3` in their initial state, the guard would still use `5`. The state key is documented as "Application-scoped max recursion depth" but only serves observability (sqlite_tracing, repl_capture) — it does not control the actual depth limit.
  - Fix: In `orchestrator.py`, read the state value and pass it: `max_depth=ctx.session.state.get("app:max_depth", 5)` in the `create_dispatch_closures()` call at line 279. Or, read it inside `create_dispatch_closures` from `ctx.session.state` before the env var override.

- **Bug: Test plan uses wrong constructor parameter name**
  - Location: Test plan Part B, `DispatchConfig(main_model='fake-model', ...)`
  - Severity: **cosmetic** (test plan artifact, not production code)
  - Description: `DispatchConfig.__init__` accepts `default_model` (dispatch.py:92), not `main_model`. The test would fail with `TypeError: DispatchConfig.__init__() got an unexpected keyword argument 'main_model'`.
  - Fix: Change to `DispatchConfig(default_model='fake-model', other_model='fake-model')`.

### Validation Notes

Static analysis confirms that `APP_MAX_DEPTH` is properly defined as a constant, included in `EXPOSED_STATE_KEYS`, set in all replay fixtures, and consumed by two observability plugins (`sqlite_tracing.py`, `repl_capture_plugin.py`). The depth guard logic in `dispatch.py:281` is correctly implemented — it returns a `DEPTH_LIMIT` error `LLMResult` when `depth + 1 >= max_depth`, and the `RLM_MAX_DEPTH` env var override at line 140 takes precedence over the parameter default. The key deficiency is that the orchestrator never bridges the session state value into the dispatch parameter, making `app:max_depth` an observability-only key rather than the runtime control it's documented to be. This disconnect is currently masked because both the hardcoded default and all fixture values are `5`. The test plan's behavioral tests (B1/B2) would validate the depth guard mechanics correctly (modulo the constructor typo) but would not detect this state-key wiring gap since they pass `max_depth` directly as a parameter.
